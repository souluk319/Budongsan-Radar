집집 데이터 전략

크롤러가 아니라, 큐레이션과 공공데이터로 간다

집집은 언론사 기사 전문을 긁어오는 서비스가 아니다. RSS, 관리자 큐레이션, URL 메타데이터, 공공데이터 API를 모아 AI가 사용자 상황별 의미를 해석하는 부동산 브리핑 서비스로 설계한다.

작성일 2026-06-13 업데이트: 2026-06-14
기준 문서: 데이터 수집/연동 준비
수집 방식

RSS, 공개 API, URL 메타데이터, 관리자/사용자 제보 중심.

금지선

언론사 기사 전문 저장, 무단 본문 크롤링, 자동 publish 금지.

확보된 키

DATA_GO_KR_SERVICE_KEY와 REB 통계 API 값은 들어왔다.

현재 env 반영 상태

값은 출력하지 않고 존재 여부만 확인했다. REB는 R-ONE Open API 기준으로 정리한다.

2026-06-14

들어온 값

  • DATA_GO_KR_SERVICE_KEY
  • REB_STATS_API_BASE_URL
  • REB_STATS_API_DOC_URL
  • REB_STATS_API_KEY

아직 받을 값

  • ECOS_API_KEY
  • NAVER_CLIENT_ID
  • NAVER_CLIENT_SECRET
  • LAW_OPEN_API_OC
실제 .env는 이제 정식 이름으로 정리됐다. 코드는 이전 로컬 env 호환을 위해 REB_URL, REB_API도 alias로 읽지만, 앞으로는 REB_STATS_API_BASE_URL, REB_STATS_API_DOC_URL, REB_STATS_API_KEY만 사용한다.

성욱이 지금 할 일

계정/본인인증/API 신청은 내가 대신할 수 없다. 여기만 먼저 끝내면 다음 구현 루프가 바로 열린다.

우선순위 1
1

공공데이터포털 키 확인

공공데이터포털에서 계정 준비 후 아래 API 활용신청을 한다. 발급 키는 이미 DATA_GO_KR_SERVICE_KEY에 들어온 것으로 확인했다.

2

필수 API 활용신청

실거래가, 전월세, 청약홈 분양정보, 청약 경쟁률, 한국부동산원 부동산통계 조회 API를 신청한다.

3

남은 키 준비

금리/거시지표는 ECOS_API_KEY, 이슈 발견은 네이버 뉴스 검색 API 키를 쓰면 된다. 법령 보강은 LAW_OPEN_API_OC를 쓴다.

4

초기 출처 10개 선정

초반 품질은 자동화보다 출처 선정이 중요하다. 믿고 볼 언론/리포트/RSS 후보를 골라두면 운영 품질이 빨리 올라간다.

역할 분리

내가 할 수 있는 것과 성욱이 직접 해야 하는 것을 명확히 나눈다.

작업 경계

내가 바로 할 수 있는 것

  • RSS allowlist 설계
  • rss_sources seed 추가
  • RSS ingest route 개선
  • canonical URL 중복 제거
  • URL 메타데이터 수집
  • pending/admin 승인 흐름 정리
  • OpenAI 요약 프롬프트/JSON schema 개선
  • 공공데이터 API adapter 구조 설계
  • 연결 테스트 스크립트 작성

키가 있으면 가능한 것

  • 공공데이터포털 API 실제 호출 테스트
  • 실거래가/청약/통계 데이터 정규화
  • 한국은행 ECOS 호출 테스트
  • 네이버 뉴스 검색 호출 테스트
  • OpenAI 실제 요약 생성 테스트
  • Supabase에 실제 수집 데이터 저장
  • 로컬/preview end-to-end 검증

내가 하지 말아야 하는 것

  • 성욱 계정 회원가입/본인인증 대행
  • 언론사 기사 전문 무단 크롤링
  • 약관 확인 없는 본문 저장/재배포
  • 비밀키를 문서/커밋/로그에 출력
  • 관리자 승인 없는 자동 publish

데이터 흐름

수집은 후보 생성까지만 자동화하고, 발행은 승인 흐름을 거친다.

운영 흐름
01 RSS/API/제보 URL
02 URL 정규화와 중복 제거
03 pending 링크 저장
04 관리자 검수와 AI 요약
05 published 후 홈/브리프 노출

수집 대상

초반에는 키 없이 가능한 RSS와 관리자 큐레이션을 먼저 붙이고, 공공데이터 API로 해석 근거를 보강한다.

이전 로컬 env 호환을 위해 REB_URL은 REB_STATS_API_DOC_URL alias로, REB_API는 REB_STATS_API_KEY alias로 읽지만 앞으로는 정식 이름만 사용한다. LAW_OPEN_API_OC는 법제처 Open API의 사용자 식별값/키를 저장하는 후보 이름이다. 실제 발급 방식 확인 후 이름은 조정할 수 있다.

구현 순서

한 번에 다 붙이지 않는다. 키 없이 가능한 것부터 실제 데이터 흐름을 만든다.

단계

Phase 1 RSS + 관리자 큐레이션

목표: 키 없이도 실제 이슈 후보가 쌓이는 구조 만들기.

  • RSS source allowlist 추가
  • /api/admin/rss/ingest가 allowlist만 수집
  • canonical URL 중복 방지
  • pending 링크 저장
  • admin에서 승인/거절
  • 승인된 링크에만 요약 생성

Phase 2 공공데이터 근거 보강

목표: 집집 요약이 그냥 뉴스 요약이 아니라 데이터 근거가 있는 브리핑이 되게 한다.

  • 실거래가 adapter
  • 전월세 adapter
  • 청약/경쟁률 adapter
  • R-ONE 부동산통계 adapter
  • 링크 카테고리/지역 기준 관련 데이터 첨부

Phase 3 검색 API/법령/거시지표 보강

목표: 이슈 발견 범위와 해석 품질을 넓힌다.

  • 네이버 뉴스 검색은 후보 발견용으로만 사용
  • ECOS로 금리/거시지표 첨부
  • 법제처 API로 법령/시행일 확인
  • 본문 전문 저장 없이 제목/링크/출처/요약 메타만 저장

성공 기준

데이터 양보다 운영 가능한 품질 흐름이 먼저다.

검증
  • RSS ingest 후 pending 링크가 생긴다.
  • 같은 RSS를 두 번 돌려도 중복 저장되지 않는다.
  • 비admin은 ingest/승인 불가.
  • 정책/청약/전세/매매 이슈 상세에서 관련 공공데이터 근거를 볼 수 있다.
  • AI 요약 프롬프트에 근거 데이터가 들어간다.
  • 근거와 추정이 분리되어 저장된다.
  • 검색 API 결과는 원문 링크 후보로만 저장된다.
  • 정책 이슈는 관련 법령/시행일 근거를 함께 보여줄 수 있다.

당장 다음 액션

이 순서로 가면 헛도는 시간이 제일 적다.

추천
  1. 나는 DATA_GO_KR_SERVICE_KEY, REB_STATS_API_KEY 기준으로 연결 테스트 구조를 만든다.
  2. 성욱은 ECOS_API_KEY, NAVER_CLIENT_ID, NAVER_CLIENT_SECRET, LAW_OPEN_API_OC를 받으러 간다.
  3. 나는 RSS allowlist + pending/admin 수집 흐름을 먼저 단단하게 만든다.
  4. 키가 더 들어오면 실거래/청약/R-ONE 통계/ECOS/네이버 검색 adapter를 순서대로 붙인다.
  5. 데이터 근거가 붙은 OpenAI 요약으로 넘어간다.

초반 집집의 품질은 데이터 양보다 큐레이션 품질이 더 중요하다. 많이 긁기보다 좋은 후보를 모으고, 승인하고, 짧게 해석하는 운영 흐름을 먼저 만든다.

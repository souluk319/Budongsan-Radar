# 집집 리서치 인사이트 정리

기준 문서: `docs/zipzip_part_v2.md`
작성일: 2026-06-14

## 한 줄 판단

집집은 단순 뉴스 큐레이션으로 끝나면 약하다. 초기에는 "오늘 부동산 이슈를 내 상황에 맞게 해석해주는 브리핑"으로 가되, 다음 축은 공공데이터 기반 지역·단지·건물 맥락과 전세/관리비/청약 리스크 해석으로 확장해야 한다.

## 리서치에서 가져올 핵심

### 1. 매물 플랫폼보다 데이터/맥락 웨지가 먼저다

문서는 처음부터 직거래, 전자계약, 결제, 금융을 한 번에 먹는 접근을 위험하다고 본다. 집집의 현재 방향과 맞다.

집집에 적용:

- 당장 매물 등록/거래 기능으로 가지 않는다
- 뉴스/RSS/API/제보를 모아 사용자의 의사결정 전 맥락을 정리한다
- 공공데이터를 붙여 "왜 중요한가"의 근거를 강화한다

### 2. 승부처는 데이터 보유가 아니라 데이터 정합성이다

한국 부동산 데이터는 이미 공공 시스템에 흩어져 있다. 핵심은 실거래가, R-ONE, 건축물대장, 도로명주소, 브이월드, K-apt, 청약Home, 리츠 데이터를 같은 내부 키 체계로 묶는 능력이다.

집집에 적용:

- 다음 DB 확장은 `links`보다 `regions`, `complexes`, `buildings`, `units`, `market_indicators` 쪽이 중요하다
- 주소/지번/좌표/단지 식별자/건축물대장 PK를 나중에라도 담을 수 있게 설계한다
- API adapter는 원천별 응답 모양을 UI에 바로 넘기지 말고 내부 표준 모델로 변환한다

### 3. AI는 범용 챗봇이 아니라 위험/비교/체크포인트용이어야 한다

리서치의 핵심 AI 포인트는 "대화형 검색" 자체가 아니라 전세 위험, 관리비 이상치, 유사 단지 비교, 허위매물 패턴, 상담 스크리닝 같은 특화 AI다.

집집에 적용:

- OpenAI 요약 프롬프트는 기사 요약보다 "위험/근거/불확실성/사용자별 체크포인트"를 우선한다
- 상세 페이지에는 `요약`보다 `이 이슈가 내 판단에 주는 영향`을 먼저 둔다
- 전세/월세/청약/금리 이슈에는 공식 데이터 근거를 붙인다

### 4. 전세 안전성과 월세화는 집집의 강한 제품 축이다

문서는 전세 안전성, 보증금 회수 가능성, 권리·채권 우선순위, 보증 가입 가능성, 신고/확정일자를 핵심 경쟁력으로 본다.

집집에 적용:

- `전세 세입자` 상황별 영향은 지금보다 더 중요하게 다룬다
- 전세/월세 카테고리에는 `보증`, `확정일자`, `선순위`, `전월세전환율`, `입주 물량` 같은 체크포인트를 붙인다
- 나중에 "전세 안전 체크" 도구형 콘텐츠를 별도 랜딩으로 만든다

### 5. 지역/단지 페이지가 SEO와 재방문 루틴의 핵심이 될 수 있다

리서치는 초기 GTM에서 도구형 콘텐츠와 지역 페이지를 강하게 추천한다. 예: 관리비 맵, 전세보증 안전체크, 단지별 실거래/관리비/입주연차 카드, 청약 체크리스트.

집집에 적용:

- 홈 브리핑만으로 끝내지 말고 `/regions/[slug]`, `/complexes/[id]` 계열 페이지를 다음 큰 축으로 본다
- "오늘의 이슈"는 지역/단지 페이지로 연결되어야 한다
- SEO 랜딩은 뉴스가 아니라 오래 남는 데이터 카드 중심으로 만든다

### 6. 브로커 우회보다 생산성 도구가 현실적이다

문서는 한국에서 중개사가 거래 실행의 핵심 노드라서 우회 전략보다 생산성 향상이 현실적이라고 본다.

집집에 적용:

- 지금은 브로커 기능을 만들지 않는다
- 다만 나중에 `상담 리드`, `지역 브로커 응답 속도`, `검증 배지`, `브로커 콘솔`로 확장할 여지는 남긴다
- 공개 제품 카피는 "중개 대체"처럼 보이면 안 된다

### 7. 보안/개인정보/감사로그는 나중 문제가 아니다

부동산은 주소, 계약서, 주민번호, 보증금, 위치정보가 얽히므로 개인정보 사고가 사업 존립 리스크가 된다.

집집에 적용:

- 지금 단계에서는 개인정보를 최소 수집한다
- 관리자 액션, 요약 생성, 링크 승인/반려, 수집 작업에는 감사로그를 붙일 준비가 필요하다
- 계약서/서류 업로드는 훨씬 뒤로 미룬다

## 집집 방향 수정

### 기존 방향

뉴스/RSS/API 링크를 모아 오늘의 부동산 이슈를 해석한다.

### 보정된 방향

뉴스/RSS/API를 입구로 쓰되, 최종 제품은 공공데이터 기반으로 지역·단지·건물 맥락과 사용자 상황별 리스크를 해석하는 부동산 의사결정 브리핑이 되어야 한다.

## 다음 구현 우선순위

### 1순위: 데이터 근거 붙은 브리핑

- 네이버/RSS 후보 수집
- 공공데이터/REB/ECOS/법령 health check
- 링크 상세에 `근거 데이터` 섹션 추가
- OpenAI 요약 프롬프트에 근거 데이터 주입

### 2순위: 전세/월세 안전 체크 관점 강화

- 전세/월세 이슈에 전세 세입자 체크포인트 강조
- 보증/확정일자/선순위/전월세전환율 관련 법령·통계 adapter 우선
- 홈에서 `세입자가 오늘 볼 것`을 더 선명하게 노출

### 3순위: 지역/단지 데이터 모델 준비

- `regions`
- `complexes`
- `buildings`
- `market_indicators`
- `data_observations`

초기에는 UI를 크게 만들지 않아도, DB 모델과 adapter 출력은 이 방향을 염두에 둔다.

### 4순위: 도구형 콘텐츠

- 전세보증 안전체크
- 청약 체크리스트
- 단지별 실거래/관리비 카드
- 지역별 입주 물량 브리프

## 금지 방향

- 매물 플랫폼처럼 보이기
- 직거래/전자계약/결제부터 붙이기
- 중개사 우회처럼 보이는 카피
- 공식 근거 없는 AI 예측
- 개인정보/계약서/위치정보를 준비 없이 수집하기
- 원천 API 응답을 그대로 화면에 흘리는 구조

## 제품 문장

집집은 매물을 보여주는 앱이 아니라, 정책·금리·전세·청약·지역 데이터를 엮어 "오늘 내 집 판단에 뭐가 중요한지" 알려주는 부동산 브리핑이다.

## 아키텍처 메모

- 현재 Next.js + Supabase + Vercel 구조는 리서치의 `린 MVP` 방향과 맞다
- 다음부터는 공공 데이터 adapter가 내부 표준 모델로 변환되도록 만든다
- 나중에 검색이 커지면 Postgres FTS/pgvector를 먼저 검토하고, 규모가 커질 때 OpenSearch로 간다
- 개인정보 도메인과 공개 데이터 도메인은 처음부터 분리해서 생각한다
